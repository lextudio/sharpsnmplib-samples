# SNMP Middleware Pipeline Design

This document describes the middleware pipeline used by `Samples.Engine` and how to customize it for advanced scenarios (custom authentication, policy enforcement, auditing, etc.).

## Overview

The new design is middleware-first, similar to ASP.NET Core:

1. Each incoming SNMP message is wrapped in `SnmpMessageContext`.
2. Middleware runs in registration order.
3. A middleware can:
   - continue (`await next(context)`),
   - short-circuit (return without calling `next`),
   - or prepare a response and stop the pipeline.
4. After pipeline completion, `SnmpEngine` sends `context.SnmpContext.Response` (if any).

Core interfaces and types:

1. `ISnmpMiddleware`
2. `SnmpPipelineBuilder`
3. `SnmpMessageContext`
4. `IMembershipProvider`
5. `SnmpEngine`

## Default Pipeline

The default `SnmpEngine` constructor builds this chain:

1. `UseContextFactory(users, group)`
2. `UseAuthentication(membership)`
3. `UseMessageHandler(store, trapV1Received, trapV2Received, informReceived)`

Responsibilities:

1. **ContextFactoryMiddleware**
   - Builds version-specific context (`NormalSnmpContext` or `SecureSnmpContext`).
2. **AuthenticationMiddleware**
   - Calls `IMembershipProvider.AuthenticateRequest(ISnmpContext)`.
   - Short-circuits if authentication fails.
   - For SNMPv3, can short-circuit with REPORT responses generated by secure context.
3. **MessageHandlerMiddleware**
   - Terminal processing for Get/GetNext/GetBulk/Set/Trap/Inform.
   - Reads/writes from `ObjectStore`.

## Customization Model

You can customize in two ways:

1. **Custom middleware** (`ISnmpMiddleware`) for cross-cutting behaviors:
   - source filtering,
   - auditing,
   - rate limiting,
   - request mutation/validation,
   - short-circuit policy responses.
2. **Custom membership providers** (`IMembershipProvider`) for authentication logic:
   - custom community rules,
   - tenant/environment-based auth,
   - source-aware auth,
   - staged v3 constraints.

## Order and Short-Circuit Rules

Recommended order:

1. `UseContextFactory(...)` first (required for `context.SnmpContext`).
2. pre-auth policy middleware (optional).
3. `UseAuthentication(...)`.
4. post-auth middleware (auditing, metrics, enrichment).
5. `UseMessageHandler(...)` as terminal.

Important:

1. If middleware does not call `next`, later middleware is skipped.
2. If middleware short-circuits without generating a response, the request is dropped silently.
3. If middleware generates a response (`CopyRequest`, `GenerateResponse`, etc.), it can return and let engine send that response.

## Example: Source-Aware Authentication + Write Policy Middleware

This example shows:

1. custom auth wrapper that allows only specific source IPs, then delegates to composed membership provider,
2. custom middleware that blocks `SET` requests during maintenance window with `NoAccess`,
3. wiring a fully custom pipeline into `SnmpEngine`.

```csharp
using System;
using System.Collections.Generic;
using System.Net;
using System.Threading.Tasks;
using Lextm.SharpSnmpLib;
using Lextm.SharpSnmpLib.Messaging;
using Lextm.SharpSnmpLib.Security;
using Samples.Pipeline;

public sealed class SourceAwareMembershipProvider : IMembershipProvider
{
    private readonly IMembershipProvider _inner;
    private readonly HashSet<IPAddress> _allowedSources;

    public SourceAwareMembershipProvider(IMembershipProvider inner, IEnumerable<IPAddress> allowedSources)
    {
        _inner = inner ?? throw new ArgumentNullException(nameof(inner));
        _allowedSources = new HashSet<IPAddress>(allowedSources ?? Array.Empty<IPAddress>());
    }

    public bool AuthenticateRequest(ISnmpContext context)
    {
        if (context == null)
        {
            throw new ArgumentNullException(nameof(context));
        }

        // Hard source filter before normal SNMP auth.
        if (!_allowedSources.Contains(context.Sender.Address))
        {
            return false; // short-circuit in AuthenticationMiddleware
        }

        return _inner.AuthenticateRequest(context);
    }
}

public sealed class SetMaintenanceWindowMiddleware : ISnmpMiddleware
{
    public async Task InvokeAsync(SnmpMessageContext context, Func<SnmpMessageContext, Task> next)
    {
        if (context == null)
        {
            throw new ArgumentNullException(nameof(context));
        }

        var snmp = context.SnmpContext;
        if (snmp == null)
        {
            throw new InvalidOperationException("SnmpContext is not initialized.");
        }

        var isSet = snmp.Request.Pdu().TypeCode == SnmpType.SetRequestPdu;
        var isMaintenanceWindow = DateTime.UtcNow.Hour >= 2 && DateTime.UtcNow.Hour < 4;
        if (isSet && isMaintenanceWindow)
        {
            // Explicit deny: prepare response and stop.
            snmp.CopyRequest(ErrorCode.NoAccess, 1);
            return;
        }

        await next(context).ConfigureAwait(false);
    }
}

public static class SnmpEngineFactory
{
    public static SnmpEngine Create(
        Listener listener,
        EngineGroup group,
        ObjectStore store,
        IEnumerable<IPAddress> allowedSources)
    {
        var baseMembership = new ComposedMembershipProvider(new IMembershipProvider[]
        {
            new Version1MembershipProvider(new OctetString("public"), new OctetString("private")),
            new Version2MembershipProvider(new OctetString("public"), new OctetString("private")),
            new Version3MembershipProvider()
        });

        var membership = new SourceAwareMembershipProvider(baseMembership, allowedSources);

        var pipeline = new SnmpPipelineBuilder()
            .UseContextFactory(listener.Users, group)
            .UseAuthentication(membership)
            .Use(new SetMaintenanceWindowMiddleware())
            .UseMessageHandler(store)
            .Build();

        return new SnmpEngine(listener, pipeline);
    }
}
```

## Practical Tips

1. Keep authentication/authorization decisions in middleware/provider, not in MIB object code.
2. Put metrics/audit middleware after authentication if you only want accepted traffic.
3. Put coarse-grained network filtering before authentication to reduce overhead.
4. Keep `UseMessageHandler` last unless you intentionally replace request handling.

## Profile Examples: v1/v2-Only and v3-Only

### v1/v2-Only Pipeline

Use only v1/v2 membership providers. v3 requests will fail authentication and be dropped.

```csharp
var membership = new ComposedMembershipProvider(new IMembershipProvider[]
{
    new Version1MembershipProvider(new OctetString("public"), new OctetString("private")),
    new Version2MembershipProvider(new OctetString("public"), new OctetString("private"))
});

var pipeline = new SnmpPipelineBuilder()
    .UseContextFactory(listener.Users, group)
    .UseAuthentication(membership)
    .UseMessageHandler(store)
    .Build();

var engine = new SnmpEngine(listener, pipeline);
```

If you want explicit behavior (for example metrics/logging on denied v3), add middleware before auth:

```csharp
public sealed class DropV3Middleware : ISnmpMiddleware
{
    public Task InvokeAsync(SnmpMessageContext context, Func<SnmpMessageContext, Task> next)
    {
        if (context.Request.Version == VersionCode.V3)
        {
            return Task.CompletedTask; // explicit drop
        }

        return next(context);
    }
}
```

### v3-Only Pipeline

Use only `Version3MembershipProvider`. v1/v2 requests will fail authentication.

```csharp
var membership = new ComposedMembershipProvider(new IMembershipProvider[]
{
    new Version3MembershipProvider()
});

var pipeline = new SnmpPipelineBuilder()
    .UseContextFactory(listener.Users, group)
    .UseAuthentication(membership)
    .UseMessageHandler(store)
    .Build();

var engine = new SnmpEngine(listener, pipeline);
```

You can pair this with a stricter source-aware provider (shown above) to lock v3 to a management subnet only.

## Drop Specific Message Types (Example: SET)

You can block selected PDU types with custom middleware.

```csharp
public sealed class BlockSetRequestsMiddleware : ISnmpMiddleware
{
    private readonly bool _silentDrop;

    public BlockSetRequestsMiddleware(bool silentDrop = false)
    {
        _silentDrop = silentDrop;
    }

    public Task InvokeAsync(SnmpMessageContext context, Func<SnmpMessageContext, Task> next)
    {
        var snmp = context.SnmpContext;
        if (snmp == null)
        {
            throw new InvalidOperationException("SnmpContext is not initialized.");
        }

        if (snmp.Request.Pdu().TypeCode == SnmpType.SetRequestPdu)
        {
            if (_silentDrop)
            {
                return Task.CompletedTask; // drop with no response
            }

            snmp.CopyRequest(ErrorCode.NoAccess, 1); // explicit denial
            return Task.CompletedTask;
        }

        return next(context);
    }
}
```

Register it before `UseMessageHandler(...)`:

```csharp
var pipeline = new SnmpPipelineBuilder()
    .UseContextFactory(listener.Users, group)
    .UseAuthentication(membership)
    .Use(new BlockSetRequestsMiddleware(silentDrop: false))
    .UseMessageHandler(store)
    .Build();
```

## Mapping from Legacy Pipeline Thinking

If you previously customized by adding/removing handler classes and mapping tables, the equivalent in middleware is:

1. **old handler mapping rules** -> **middleware order + branching logic**
2. **old auth hooks** -> **`IMembershipProvider` + auth middleware**
3. **old pre/post handler hooks** -> **middleware before/after `UseMessageHandler`**

The net effect is the same level of customization, with clearer ordering and composability.
